# 個別レース分析フロー

このスキルは `/analyze <race_id>` から起動される。メインClaude が司会・統合役を担い、サブエージェントを選抜呼び出しして判断材料を集める。

## ステップ1: 入力解析とデータ準備

1. `race_id` を正規化（keibalab 日付ベース形式、12桁 = YYYYMMDD+場コード+R。例 `202605030811`）
2. `data/race.db` の `races` テーブルから `course`, `distance`, `surface`, `grade`, `class` を取得
3. プレイブックキー（`<course>-<distance>`）を判定
4. **不変データの鮮度チェック**（一度取れば変わらない情報だけ DB に取得）：
   - `python scripts/fetch_races.py <race_id>` を実行（出走馬の馬番・斤量・騎手など出馬投票後の確定情報）
   - 出走馬の血統データが古ければ `python scripts/fetch_pedigree.py --race <race_id>`
   - 出走馬の過去走が古ければ `python scripts/fetch_results.py --horses-from-race <race_id>`
5. **揮発データは取得しない**（オッズ・馬体重・馬場発表・天候は macro-scout がステップ3で WebFetch する）
6. **対象馬リスト確定（全頭分析の起点）**: `entries` テーブルから出馬表全頭（通常 12-18 頭）を取得し、馬番順に並べた **対象馬リスト** を作る。これ以降のすべてのエージェント呼び出しで、このリストの **全頭** を評価対象として明示的に渡す。出馬表が取得できていない場合は **scout に WebFetch で先に確定させる**（部分情報での分析は最終手段）。

## ステップ2: エージェント選抜

**デフォルト構成**（軽量モード）:
- `pedigree-analyst`（必須）
- `macro-scout`（必須・揮発データ＝オッズ・馬体重・馬場の取得担当）
- `devils-advocate`（必須・最後）

**追加招集の判断**:
- ユーザーが「位置取り」「脚質」「コース適性」「持ち時計」を含意 → `track-analyst` 追加
- レースが重賞（GI/GII/GIII）または該当プレイブックがある → `race-context-analyst` 追加
- ユーザーが `--full` を指定 → 全員招集
- ユーザーが `--quick` を指定 → pedigree + macro-scout + devils-advocate のみ厳守

> **重要**: `macro-scout` は軽量モードでも必ず呼ぶ。理由: オッズ・馬体重・馬場発表は DB に入っておらず、devils-advocate のオッズ妙味判定や直前情報チェックに必須。`--quick` の場合、scout の取得範囲を「オッズ・当日馬体重・馬場発表」に限定してコストを抑える。

### 全頭分析必須ルール（ハードルール）

呼び出されたすべての専門エージェント（pedigree / track / race-context / devils-advocate）は **対象馬リストの全頭** を評価対象とする。「上位人気数頭だけ」「注目馬だけ」のような暗黙の絞り込みは禁止。理由：

1. 穴馬・伏兵の見落としを構造的に防ぐ
2. ◎○▲△▽の選定は **全頭評価の上での絞り込み** であり、評価しない馬を消し扱いするのは恣意的
3. 評論家（devils-advocate）が「埋もれている評価馬」を拾うには、全頭評価が前提

各エージェントには **対象馬リスト（馬番・馬名・horse_id）を全頭** 渡し、出力に **全頭評価サマリ表** を含めるよう明示すること。コンテキストを膨らませないため、深掘り記述は強気度上位 5-7 頭＋消し評価上位 1-2 頭に絞ってよいが、**全頭の強気度・確信度・キー論点 1 行は必須**。

### データ不足時の扱い

出馬表の一部しか取得できない緊急時は、対応順に：
1. 取得を再試行（scout に WebFetch 追加依頼、または手動補完を要請）
2. それでも不足するなら、**取得済み頭数のみで「擬似全頭」評価**を行い、未取得馬を出力に明示（「N頭未取得・買い目から除外」）
3. 部分情報で完了する場合は、出力冒頭に **データ取得状況** セクションを置き、評価のリスクを警告

## ステップ3: エージェント実行

1. **scout を先に / もしくは並列で起動**: scout の出力（特にオッズ・馬体重・馬場）は他エージェントの分析の前提になる
2. pedigree / track / race-context を **並列** で呼び出し（独立した分析なので並列化）。データソースは `data/race.db`（不変データ）
3. 各エージェントに渡す: race_id、コース・距離・グレード（判定済み）、scout の出力（揮発データ）、関連データソースの場所、**対象馬リスト（馬番順の全頭）**、**「全頭評価必須」の明示**
4. 全員の結論が揃ったら、それを `devils-advocate` に渡して最後に呼ぶ
   - 評論家には他エージェントの **強気度・根拠・反証条件** と scout の揮発データ をすべて見せる
   - 評論家には「他エージェントが過小評価している馬を全頭評価表から拾う」よう明示

## ステップ5: ガード

- データの出所（DB / netkeiba 直接 / WebFetch）を必ず明記
- 古いデータを使う場合は警告
- **賭博助言ではない** ことを末尾に1行付記
- 評論家の弱気論を「平均化」して薄めない（メイン統合時のバイアス）
- **オッズ妙味** を必ず1点はチェック（想定勝率 × オッズ ＝ 期待値）

## 分析モードの判定（引数の形で決まる）

watchlist は廃止。**対象は都度プロンプト/引数で指定**する。分析モードは引数の形で判定し、論点とアウトプットを調整する。

### 1. レース分析（`/analyze <race_id>` または keibalab レースURL）
- そのレースの **買い目・軸馬選定** が中心論点
- race_id は keibalab 日付ベース 12桁（`YYYYMMDD+場コード+R`、例 `202605030811`）。`https://www.keibalab.jp/db/race/<race_id>/...` を貼られた場合は ID を抽出

### 2. 馬分析（`/analyze --horse <horse_id>` または keibalab 馬URL）
- 特定馬の **適性・次走分析** モード
- 出走予定があれば、そのレースを連動分析
- horse_id は 10桁。`https://www.keibalab.jp/db/horse/<horse_id>/` を貼られた場合は ID を抽出

対象が DB に無ければ、先に `fetch_*.py`（不変データ）または scout の WebFetch（揮発データ）で取得してから分析する。
